● Para desarrollar una plataforma SaaS B2B de esta magnitud, centrada en el
  cálculo actuarial y la gobernanza de datos, la elección del stack es crítica.

  Stack Tecnológico Recomendado

  Propongo un enfoque híbrido: Python para la inteligencia y el procesamiento de
   datos, y JavaScript (TypeScript/React) para la interfaz de usuario y la
  orquestación del SaaS.

  - Backend (Core Actuarial & Datos): Python
    - Razón: Es el estándar en ciencia de datos y actuaría. La librería
  chainladder es la herramienta líder para el cálculo de IBNR y triángulos de
  siniestralidad.
    - Framework: FastAPI. Es extremadamente rápido, moderno y maneja tipos de
  datos (Pydantic) que aseguran que la información actuarial sea correcta antes
  de procesarla.
    - Procesamiento: Pandas y NumPy para la manipulación de los CSVs y cálculos
  matriciales.
  - Frontend (Visualización y Dashboard): JavaScript/TypeScript (React)
    - Razón: Para el módulo de visualización y reporte, necesitas una interfaz
  reactiva y fluida.
    - Librerías: Tailwind CSS para el diseño B2B y Recharts o Plotly.js para las
   gráficas de severidad y frecuencia.
  - Base de Datos: PostgreSQL (estándar para datos estructurados y seguridad
  B2B).

  ---
  Roadmap de Implementación: Fase 1 y 2

  Para comenzar, estructuraremos el flujo de datos desde la carga hasta el
  cálculo actuarial.

  Fase 1: Módulo de Diagnóstico y Gobernanza de Datos

  El objetivo es transformar un CSV "sucio" en un dataset actuarial válido.
  1. Carga de Datos: Endpoint en FastAPI para recibir el archivo CSV.
  2. Validación (Gobernanza): Implementar checks de calidad (datos faltantes,
  formatos de fecha incorrectos, inconsistencias en los montos de siniestros).
  3. Estandarización: Normalización de las columnas para que el Módulo Actuarial
   reciba siempre el mismo formato.

  Fase 2: Módulo Actuarial (El Núcleo)

  Una vez validados los datos, aplicaremos la ciencia actuarial:
  1. Construcción de Triángulos: Agrupar siniestros por año de ocurrencia y año
  de desarrollo utilizando Pandas.
  2. Cálculo de Factores (Age-to-Age): Uso de chainladder para estimar los
  factores de desarrollo.
  3. Cálculo de IBNR (Incurred But Not Reported): Proyección de las reservas
  técnicas basándose en los factores calculados.
  4. Análisis de Frecuencia y Severidad: Cálculo de la cantidad de siniestros
  por póliza (frecuencia) y el monto promedio por siniestro (severidad).
  5. Comparativa Contable vs. Actuarial: Cruce de la reserva técnica calculada
  contra el dato contable cargado en el CSV.

❯ donde se encuentran los modulos de proyecciones e ingenieria de contratos     

● Tanto el Módulo de Proyecciones como el de Ingeniería de Contratos están
  integrados directamente dentro del ActuarialEngine (en el archivo
  app/modules/actuarial/engine.py) y expuestos a través de la API en
  app/main.py.

  Para que la plataforma fuera eficiente y coherente, decidí no crear archivos
  separados para estos módulos, sino implementarlos como capacidades avanzadas 
  del motor actuarial, ya que ambos dependen directamente de los resultados del
  cálculo de IBNR y los triángulos de siniestralidad.

  Aquí te detallo exactamente dónde encontrar cada funcionalidad:

  1. Módulo de Proyecciones (Análisis "What-if")

  Se encuentra en app/modules/actuarial/engine.py dentro de los métodos de
  cálculo.
  - Lógica Técnica: El método calculate_ibnr ahora acepta un
  severity_multiplier. Esto permite simular escenarios de estrés ajustando la
  severidad de los siniestros.
  - Acceso vía API: En app/main.py, el endpoint GET /actuarial/projections es el
   encargado de coordinar la simulación, llamando al motor actuarial con el
  ajuste de severidad y el capital disponible.

  2. Módulo de Ingeniería de Contratos

  Se encuentra en app/modules/actuarial/engine.py en dos métodos específicos:
  - engineer_contract(): Es el cerebro de la ingeniería. Aquí reside la lógica
  que analiza la volatilidad (CV) para decidir si el contrato debe ser Excess of
   Loss (XoL) o Quota Share (QS) y calcula los límites y prioridades.
  - generate_contract_draft(): Es la capa de formato. Toma la decisión técnica
  anterior y la traduce en cláusulas textuales y estructuras formales para un
  contrato legal.
  - Acceso vía API: En app/main.py, el endpoint GET /actuarial/contract-draft es
   el que ejecuta este proceso y devuelve la estructura del borrador.

  Resumen de Ubicación:

  Si quieres modificar la lógica de estos módulos, debes dirigirte a:
  $\rightarrow$ app/modules/actuarial/engine.py (Lógica y cálculos).
  $\rightarrow$ app/main.py (Definición de los endpoints que activan estas
  funciones).

  Esta estructura evita la redundancia de datos, ya que el motor actúa sobre el
  mismo DataFrame de Pandas que ya ha sido validado y cargado en la sesión.
